#+title: fstart Implementation Plan: Scoped Ideas with Concrete Guidance
#+date: 2025-12-14

* Overview

This document takes the ideas from [[file:plan_prompt.org]] and [[file:gpt5_2_plan.org]], critiques them, and provides concrete implementation guidance with specific crate recommendations, code snippets, and considerations for porting AST2600 (Aspeed BMC SoC) from U-Boot to fstart.

*Key decision*: RON is the sole configuration format. FDT is fully discarded. Any future FDT-to-RON translation tooling is out of scope for this document.

* Critique of Original Ideas

** RON as Sole Configuration Format

*** Original Proposal (plan_prompt.org)
Use RON because it maps 1:1 to Rust types.

*** Analysis
This is the right call. The advantages:

1. *Type safety at build time*: RON deserializes directly into Rust structs. Errors are caught immediately.
2. *Human-friendly authoring*: RON is only for humans to read and write. Clean syntax, comments, readable diffs.
3. *Build tool works with native structs*: Once RON is parsed, the build tool operates on Rust structs directly - no further serialization needed.
4. *Firmware gets binary blob*: Only firmware needs to deserialize (via postcard) when runtime config is required.
5. *Composition via Rust*: Instead of FDT overlays or includes, use Rust's module system and serde's flatten/default mechanisms.
6. *Strongly typed references*: Instead of FDT phandles (opaque u32), use Rust enums or string identifiers that the compiler validates.

*** What We Lose (and Why It's OK)
- *Existing DTS files*: Must be manually translated. This is one-time work per SoC.
- *dtc tooling*: Replaced by cargo + fstart-tool.

** Device Graph: Native Rust Types

*** Original Proposal (gpt5_2_plan.org)
Define a minimal, fstart-specific device graph in RON with strongly-typed NodeKind enums.

*** Fully Endorsed
This is correct. The device graph should be:
- Strongly typed Rust enums for node kinds
- Explicit typed references (not phandles)
- Validated at build time

** Dependency-Driven Boot Flow

*** Original Proposal
Build a DAG of driver init tasks, topologically sort, iterate.

*** Critique
Sound, but with native Rust types we can do better than runtime DAG resolution:

1. *Option A: Explicit ordering in config* - User specifies init order
2. *Option B: Build-time DAG resolution* - Tool computes order, generates static array
3. *Option C: Type-system enforced* - Services are typed; can't call UART without ClockService existing
4. *Option D: Async with implicit ordering* - Generate async init functions that await dependencies

*** Recommendation: Option D - Async with Minimal Executor

Instead of computing topological order at build time, generate async init functions where dependencies are expressed as =.await= calls. A minimal executor polls all futures until done - no sleeping, no interrupts, just spin until dependencies resolve.

*Benefits*:
- Simpler codegen (no topological sort)
- Dependencies explicit in generated code
- Natural parallelism where possible
- Easier to debug (can see what's waiting on what)

*Implementation*: Minimal executor that never sleeps - just polls futures in a loop:

#+begin_src rust
//! fstart-executor/src/lib.rs - Minimal async executor for boot

#![no_std]

use core::future::Future;
use core::pin::Pin;
use core::task::{Context, Poll, RawWaker, RawWakerVTable, Waker};

/// Noop waker - we poll in a loop, never actually wake
fn noop_waker() -> Waker {
    const VTABLE: RawWakerVTable = RawWakerVTable::new(
        |_| RawWaker::new(core::ptr::null(), &VTABLE), // clone
        |_| {},  // wake
        |_| {},  // wake_by_ref
        |_| {},  // drop
    );
    unsafe { Waker::from_raw(RawWaker::new(core::ptr::null(), &VTABLE)) }
}

/// Run multiple futures to completion by polling in a loop
/// No sleeping - pure spin until all done
pub fn block_on_all<const N: usize>(
    mut futures: [Pin<&mut dyn Future<Output = Result<(), Error>>>; N]
) -> Result<(), Error> {
    let waker = noop_waker();
    let mut cx = Context::from_waker(&waker);
    
    let mut done = [false; N];
    let mut results = [None; N];
    
    while !done.iter().all(|&d| d) {
        for i in 0..N {
            if !done[i] {
                match futures[i].as_mut().poll(&mut cx) {
                    Poll::Ready(result) => {
                        done[i] = true;
                        results[i] = Some(result);
                    }
                    Poll::Pending => {}
                }
            }
        }
    }
    
    // Check for errors
    for result in results.into_iter().flatten() {
        result?;
    }
    Ok(())
}
#+end_src

*Service registry* - futures poll until their dependency is published:

#+begin_src rust
//! Service registry for dependency resolution

use core::cell::UnsafeCell;
use core::future::Future;
use core::marker::PhantomData;
use core::pin::Pin;
use core::task::{Context, Poll};

/// Maximum number of services (devices) - sized for typical SoC
pub const MAX_SERVICES: usize = 64;

/// Registry of initialized services
pub struct ServiceRegistry {
    /// Pointers to services, None if not yet initialized
    services: UnsafeCell<[Option<*const ()>; MAX_SERVICES]>,
}

// Safety: single-threaded boot environment
unsafe impl Sync for ServiceRegistry {}

impl ServiceRegistry {
    pub const fn new() -> Self {
        Self {
            services: UnsafeCell::new([None; MAX_SERVICES]),
        }
    }
    
    /// Publish a service once initialized
    pub fn publish<T>(&self, id: u16, service: &T) {
        unsafe {
            (*self.services.get())[id as usize] = Some(service as *const T as *const ());
        }
    }
    
    /// Get a future that resolves when service is available
    pub fn get<T>(&self, id: u16) -> GetService<'_, T> {
        GetService {
            registry: self,
            id,
            _phantom: PhantomData,
        }
    }
    
    /// Try to get service immediately (non-async)
    pub fn try_get<T>(&self, id: u16) -> Option<&T> {
        unsafe {
            (*self.services.get())[id as usize]
                .map(|ptr| &*(ptr as *const T))
        }
    }
}

/// Future that polls until a service is available
pub struct GetService<'a, T> {
    registry: &'a ServiceRegistry,
    id: u16,
    _phantom: PhantomData<T>,
}

impl<T> Future for GetService<'_, T> {
    type Output = &T;
    
    fn poll(self: Pin<&mut Self>, _cx: &mut Context<'_>) -> Poll<Self::Output> {
        match self.registry.try_get::<T>(self.id) {
            Some(service) => Poll::Ready(service),
            None => Poll::Pending,
        }
    }
}
#+end_src

*Generated init code* - each device gets an async function:

#+begin_src rust
//! Generated by fstart-tool
//! stages/spl/src/generated.rs

use fstart_executor::{block_on_all, ServiceRegistry};
use fstart_soc_ast2600::{scu, uart, sdram, spi};

// Device indices (from flattened hierarchy)
mod dev {
    pub const SCU: u16 = 0;
    pub const UART5: u16 = 1;
    pub const SDRAM: u16 = 2;
    pub const FMC: u16 = 3;
    pub const FLASH0: u16 = 4;
}

static SERVICES: ServiceRegistry = ServiceRegistry::new();

async fn init_scu() -> Result<(), Error> {
    // No dependencies - init immediately
    let service = scu::init(&scu::Config {
        revision: scu::Revision::A3,
        cpu_freq_mhz: 800,
    })?;
    SERVICES.publish(dev::SCU, &service);
    Ok(())
}

async fn init_uart5() -> Result<(), Error> {
    // Wait for SCU (clock controller)
    let scu = SERVICES.get::<scu::ClockService>(dev::SCU).await;
    
    let service = uart::init(&uart::Config {
        id: uart::Id::Uart5,
        baud: 115200,
    }, scu)?;
    SERVICES.publish(dev::UART5, &service);
    
    log::info!("Console ready");
    Ok(())
}

async fn init_sdram() -> Result<(), Error> {
    // Wait for SCU
    let scu = SERVICES.get::<scu::ClockService>(dev::SCU).await;
    
    let service = sdram::init(&sdram::Config {
        size_mb: 512,
        kind: sdram::DramKind::Ddr4_2400,
    }, scu)?;
    SERVICES.publish(dev::SDRAM, &service);
    
    log::info!("DRAM initialized: 512 MB");
    Ok(())
}

async fn init_fmc() -> Result<(), Error> {
    // Wait for SCU
    let scu = SERVICES.get::<scu::ClockService>(dev::SCU).await;
    
    let service = spi::init_controller(&spi::ControllerConfig {
        base: 0x1e620000,
        max_speed_hz: 50_000_000,
    }, scu)?;
    SERVICES.publish(dev::FMC, &service);
    Ok(())
}

async fn init_flash0() -> Result<(), Error> {
    // Wait for parent (FMC controller)
    let fmc = SERVICES.get::<spi::Controller>(dev::FMC).await;
    
    let service = spi::init_flash(&spi::FlashConfig {
        cs: 0,
        size_bytes: 0x4000000,
    }, fmc)?;
    SERVICES.publish(dev::FLASH0, &service);
    Ok(())
}

/// Main boot entry point
pub fn run_boot() -> Result<(), Error> {
    // Pin futures on stack
    let mut f_scu = core::pin::pin!(init_scu());
    let mut f_uart5 = core::pin::pin!(init_uart5());
    let mut f_sdram = core::pin::pin!(init_sdram());
    let mut f_fmc = core::pin::pin!(init_fmc());
    let mut f_flash0 = core::pin::pin!(init_flash0());
    
    // Run all - executor handles ordering via dependency awaits
    block_on_all([
        f_scu.as_mut() as Pin<&mut dyn Future<Output = _>>,
        f_uart5.as_mut(),
        f_sdram.as_mut(),
        f_fmc.as_mut(),
        f_flash0.as_mut(),
    ])?;
    
    // All services ready - load next stage
    let flash = SERVICES.try_get::<spi::Flash>(dev::FLASH0).unwrap();
    load_next_stage(flash)?;
    
    Ok(())
}
#+end_src

The executor just spins, polling all futures. When =init_flash0= polls, it returns =Pending= until =init_fmc= completes and publishes the FMC service. No explicit ordering needed in config or codegen - dependencies are naturally expressed as awaits.

*Why not full embassy?*
- Embassy includes timers, interrupts, wakers that actually wake
- We don't need any of that - boot is single-threaded, no sleeping
- Our executor is ~50 lines, embassy-executor is much larger
- If we later need timers (e.g., DRAM training timeouts), we can add targeted support

** Filesystem Descriptor Format

*** Original Proposal (gpt5_2_plan.org)
Use =serde-brief= for a compact self-describing format.

*** Analysis
Since we use serde throughout, the same Rust types used for RON configs can be serialized to any serde-compatible binary format. The firmware doesn't parse RON - it parses whatever binary format we choose.

Self-describing format options:

| Format         | Crate            | Size overhead | Parse complexity | no_std | Self-describing |
|----------------+------------------+---------------+------------------+--------+-----------------|
| postcard       | =postcard=       | Minimal       | Very simple      | Yes    | With schema     |
| CBOR           | =minicbor=       | Low           | Moderate         | Yes    | Yes             |
| MessagePack    | =rmp-serde=      | Low           | Moderate         | Yes*   | Partial         |
| bincode        | =bincode=        | Minimal       | Simple           | Yes*   | No              |

*** Recommendation: postcard for Firmware
=postcard= is the best fit for firmware-side deserialization:
- Designed for embedded/no_std from the start
- Minimal overhead
- =postcard-schema= crate provides self-description when needed
- Same Rust types as RON config

Layout:
1. *Header*: Fixed =zerocopy= struct (magic, version, offsets) - trivial to parse in early boot
2. *File table*: =postcard=-serialized - firmware deserializes this
3. *Embedded config* (optional): =postcard=-serialized config for runtime binding
4. *Validation*: Signatures appended after table

The build tool reads RON into structs, works with those structs directly, and only serializes to postcard when writing the final image for firmware consumption.

** Build-time vs Runtime Binding

*** Original Proposal
Support both via features.

*** Critique
For the AST2600 BMC use case, runtime binding adds complexity with little benefit. The "bulked" build mode from design_document.org is interesting for development but not critical path.

*** Recommendation: Build-time Only (Initially)
- Build tool generates all bindings as Rust code
- Runtime binding can be added later as a feature if needed
- Simplifies the mental model significantly

* Driver Config Struct Pattern

** Problem
The build tool needs to parse RON containing driver-specific configuration (UART baud rates, DRAM timings, etc.), but shouldn't need to compile actual hardware drivers (which require no_std, target-specific code).

** Solution: Feature-Gated Config Structs
Each driver crate exposes its config structs behind a =config= feature:

** Device Model: Hierarchical Config, Flat Firmware Representation

*** Design Principles
1. *RON config uses natural hierarchy*: Devices nested under their parents - intuitive for humans
2. *Firmware uses flat representation*: Build tool flattens to array + separate parent/child indices
3. *Nice firmware APIs*: Wrapper functions for =parent()=, =children()=, =ancestors()= traversal
4. *String IDs in config*: Human-readable, validated at build time
5. *Integer indices in firmware*: Efficient, no string comparisons at runtime

This gives the best of both worlds: intuitive configuration and efficient runtime.

*** RON Config Types (Hierarchical)

#[derive(Debug, Default, Deserialize, Serialize)]
pub struct DeviceConfig {

    pub devices: Vec<DeviceNode>,
    
    /// Console device ID
    #[serde(default)]
    pub console: Option<DeviceId>,
}

/// Unique device identifier (string in config)
pub type DeviceId = String;

/// A device node with optional children - used in RON config
#[derive(Debug, Clone, Deserialize, Serialize)]
pub struct DeviceNode {
    /// Unique identifier for this device
    pub id: DeviceId,
    
    /// Device type and configuration
    pub kind: DeviceKind,
    
    /// Child devices (e.g., SPI flash under SPI controller)
    #[serde(default)]
    pub children: Vec<DeviceNode>,

/// Device types - each variant contains its config
/// 
/// Three categories:
/// 1. Static devices - fully configured at build time
/// 2. Enumerable buses - bus config static, children discovered at runtime  
/// 3. Enumerated devices - matched at runtime, config specifies match criteria + driver
#[derive(Debug, Clone, Deserialize, Serialize)]
pub enum DeviceKind {
    // ========================================
    // Generic static devices (portable across SoCs)
    // ========================================
    
    /// 16550-compatible UART
    Uart(UartConfig),
    
    /// SPI controller (children are SPI devices)
    SpiController(SpiControllerConfig),
    
    /// SPI flash (child of SpiController)
    SpiFlash(SpiFlashConfig),
    
    /// I2C controller (children are I2C devices)
    I2cController(I2cControllerConfig),
    
    /// GPIO controller
    Gpio(GpioConfig),
    
    /// eMMC controller
    Emmc(EmmcConfig),
    
    // ========================================
    // AST2600-specific devices
    // ========================================
    
    /// AST2600 System Control Unit (clock, reset, pinmux)
    Ast2600Scu(Ast2600ScuConfig),
    
    /// AST2600 SDRAM controller
    Ast2600Sdmc(Ast2600SdmcConfig),
    
    /// AST2600 FMC (Firmware Memory Controller - SPI flash)
    Ast2600Fmc(Ast2600FmcConfig),
    
    /// AST2600 Hash/Crypto engine
    Ast2600Hace(Ast2600HaceConfig),
    
    // ========================================
    // Enumerable buses (static config, runtime discovery)
    // ========================================
    
    /// PCIe Root Complex - children discovered via enumeration
    PcieRootComplex(PcieRcConfig),
    
    /// USB Host Controller - children discovered via enumeration
    UsbHostController(UsbHcConfig),
    
    // ========================================
    // Enumerated devices (matched at runtime)
    // These specify match criteria; actual binding happens during bus enumeration
    // ========================================
    
    /// PCIe/NVMe device - matched by class code 0x010802
    Nvme(NvmeConfig),
    
    /// PCIe device - matched by vid:did or class
    PcieDevice(PcieDeviceConfig),
    
    /// I2C device - matched by address (or probed)
    I2cDevice(I2cDeviceConfig),
    
    /// SPI device (non-flash) - matched by CS
    SpiDevice(SpiDeviceConfig),
    
    /// TPM - can be SPI, I2C, or MMIO; matched accordingly
    Tpm(TpmConfig),
    
    /// USB device - matched by vid:pid or class
    UsbDevice(UsbDeviceConfig),

    // Fallback
    // ========================================
    
    /// Generic device for platform-specific or unknown devices
    Generic(GenericDeviceConfig),
}

    use Platform::*;

        // Generic devices valid everywhere
        (Uart(_) | SpiController(_) | SpiFlash(_) | I2cController(_) | 
         Gpio(_) | Emmc(_) | PcieRootComplex(_) | Nvme(_) | Tpm(_), _) => Ok(()),

        // Enumerated devices valid if parent bus exists
        (PcieDevice(_) | I2cDevice(_) | SpiDevice(_) | UsbDevice(_), _) => Ok(()),
        
        _ => Ok(()),

//! fstart-driver/src/device.rs - Firmware-side device representation

/// Index into the device array
pub type DeviceIdx = u16;

/// Sentinel value for "no parent" or "no more children"
pub const NO_DEVICE: DeviceIdx = DeviceIdx::MAX;

/// Flattened device entry for firmware
#[derive(Debug, Clone)]
pub struct FlatDevice {
    /// Device ID (interned string index or small inline)
    pub id: DeviceId,
    
    /// Index of parent device, or NO_DEVICE if root
    pub parent: DeviceIdx,
    
    /// Index of first child, or NO_DEVICE if leaf
    pub first_child: DeviceIdx,
    
    /// Index of next sibling, or NO_DEVICE if last
    pub next_sibling: DeviceIdx,
    
    /// Device-specific config (enum discriminant + data)
    pub kind: DeviceKind,
}

/// Device graph - flat array with hierarchy via indices
pub struct DeviceGraph {
    devices: &'static [FlatDevice],
}

impl DeviceGraph {
    /// Get device by index
    pub fn get(&self, idx: DeviceIdx) -> Option<&FlatDevice> {
        self.devices.get(idx as usize)
    }
    
    /// Find device by ID
    pub fn find(&self, id: &str) -> Option<(DeviceIdx, &FlatDevice)> {
        self.devices.iter()
            .enumerate()
            .find(|(_, d)| d.id == id)
            .map(|(i, d)| (i as DeviceIdx, d))
    }
    
    /// Get parent of device
    pub fn parent(&self, idx: DeviceIdx) -> Option<(DeviceIdx, &FlatDevice)> {
        let dev = self.get(idx)?;
        if dev.parent == NO_DEVICE {
            None
        } else {
            Some((dev.parent, self.get(dev.parent)?))
        }
    }
    
    /// Iterate over children of device
    pub fn children(&self, idx: DeviceIdx) -> ChildIter<'_> {
        let first = self.get(idx)
            .map(|d| d.first_child)
            .unwrap_or(NO_DEVICE);
        ChildIter { graph: self, current: first }
    }
    
    /// Iterate over ancestors (parent, grandparent, etc.)
    pub fn ancestors(&self, idx: DeviceIdx) -> AncestorIter<'_> {
        AncestorIter { graph: self, current: idx }
    }
    
    /// Check if `ancestor_idx` is an ancestor of `idx`
    pub fn is_ancestor(&self, idx: DeviceIdx, ancestor_idx: DeviceIdx) -> bool {
        self.ancestors(idx).any(|(i, _)| i == ancestor_idx)
    }
}

/// Iterator over children of a device
pub struct ChildIter<'a> {
    graph: &'a DeviceGraph,
    current: DeviceIdx,
}

impl<'a> Iterator for ChildIter<'a> {
    type Item = (DeviceIdx, &'a FlatDevice);
    
    fn next(&mut self) -> Option<Self::Item> {
        if self.current == NO_DEVICE {
            return None;
        }
        let dev = self.graph.get(self.current)?;
        let idx = self.current;
        self.current = dev.next_sibling;
        Some((idx, dev))
    }
}

/// Iterator over ancestors of a device
pub struct AncestorIter<'a> {
    graph: &'a DeviceGraph,
    current: DeviceIdx,
}

impl<'a> Iterator for AncestorIter<'a> {
    type Item = (DeviceIdx, &'a FlatDevice);
    
    fn next(&mut self) -> Option<Self::Item> {
        let dev = self.graph.get(self.current)?;
        if dev.parent == NO_DEVICE {
            return None;
        }
        self.current = dev.parent;
        let parent = self.graph.get(self.current)?;
        Some((self.current, parent))
    }
}

*** Build Tool: Flattening Hierarchy

#+begin_src rust
//! fstart-tool/src/flatten.rs - Convert hierarchical config to flat representation

use fstart_config::{DeviceConfig, DeviceNode};

pub struct FlattenedDevices {
    pub devices: Vec<FlatDevice>,
    pub id_to_idx: HashMap<String, u16>,
}

pub fn flatten_devices(config: &DeviceConfig) -> FlattenedDevices {
    let mut devices = Vec::new();
    let mut id_to_idx = HashMap::new();
    
    // First pass: assign indices via depth-first traversal
    fn collect_indices(
        node: &DeviceNode,
        id_to_idx: &mut HashMap<String, u16>,
        counter: &mut u16,
    ) {
        id_to_idx.insert(node.id.clone(), *counter);
        *counter += 1;
        for child in &node.children {
            collect_indices(child, id_to_idx, counter);
        }
    }
    
    let mut counter = 0u16;
    for root in &config.devices {
        collect_indices(root, &mut id_to_idx, &mut counter);
    }
    
    // Second pass: build flat devices with parent/child/sibling indices
    fn build_flat(
        node: &DeviceNode,
        parent_idx: u16,
        next_sibling_idx: u16,
        id_to_idx: &HashMap<String, u16>,
        devices: &mut Vec<FlatDevice>,
    ) {
        let my_idx = id_to_idx[&node.id];
        
        // Calculate first_child index
        let first_child = node.children.first()
            .map(|c| id_to_idx[&c.id])
            .unwrap_or(NO_DEVICE);
        
        devices.push(FlatDevice {
            id: node.id.clone(),
            parent: parent_idx,
            first_child,
            next_sibling: next_sibling_idx,
            kind: node.kind.clone(),
        });
        
        // Process children with correct sibling links
        for (i, child) in node.children.iter().enumerate() {
            let next_sib = node.children.get(i + 1)
                .map(|c| id_to_idx[&c.id])
                .unwrap_or(NO_DEVICE);
            build_flat(child, my_idx, next_sib, id_to_idx, devices);
        }
    }
    
    // Process root nodes
    for (i, root) in config.devices.iter().enumerate() {
        let next_sib = config.devices.get(i + 1)
            .map(|r| id_to_idx[&r.id])
            .unwrap_or(NO_DEVICE);
        build_flat(root, NO_DEVICE, next_sib, &id_to_idx, &mut devices);
    }
    
    // Sort by index for correct ordering
    devices.sort_by_key(|d| id_to_idx[&d.id]);
    
    FlattenedDevices { devices, id_to_idx }
}

*** Runtime Enumeration Flow

For enumerable buses (PCIe, USB), the hierarchy in config declares *expected* devices. The async init flow:

1. Bus controller init (=PcieRootComplex=, =UsbHostController=) runs first
2. Bus enumeration discovers actual devices on the bus
3. Discovered devices are matched against config entries by vid:did, class code, etc.
4. Matched devices get their =PciBinding= or =UsbBinding= published to the registry
5. Child device inits (=Nvme=, =PcieDevice=, =UsbDevice=) await their binding, then init

#+begin_src rust
// Generated async init for pcie0 (enumerable bus)
async fn init_pcie0(services: &ServiceRegistry, graph: &DeviceGraph) -> Result<(), Error> {
    // Init the root complex hardware
    let rc = pcie::init_root_complex(&CONFIG.pcie0)?;
    
    // Enumerate the bus - discover all devices
    let discovered = pcie::enumerate(&rc)?;
    
    // Match discovered devices to config entries
    let (pcie_idx, _) = graph.find("pcie0").unwrap();
    for (child_idx, child) in graph.children(pcie_idx) {
        let binding = match &child.kind {
            DeviceKind::Nvme(cfg) => {
                // NVMe always matches by class 0x010802
                discovered.find_by_class(0x010802)
            }
            DeviceKind::PcieDevice(cfg) => {
                // Match by vid:did or class
                cfg.vid_did.and_then(|(v, d)| discovered.find_by_vid_did(v, d))
                    .or_else(|| cfg.class_code.and_then(|c| discovered.find_by_class(c)))
            }
            _ => None,
        };
        
        if let Some(pci_dev) = binding {
            services.publish_pci_binding(child_idx, pci_dev);
        }
    }
    
    services.publish(dev::PCIE0, &rc);
    Ok(())
}

// Generated async init for nvme0 - waits for enumeration to bind it
async fn init_nvme0(services: &ServiceRegistry) -> Result<(), Error> {
    // Wait for pcie0's enumeration to find and bind us
    let pci_dev = services.get_pci_binding(dev::NVME0).await;
    
    let nvme = nvme::init(&CONFIG.nvme0, pci_dev)?;
    services.publish(dev::NVME0, &nvme);
    Ok(())
}

*** How Ordering Works Without Explicit Ops

The device hierarchy in config implies dependencies:
- Child devices depend on their parent
- All devices implicitly depend on the clock controller (SCU)
- Console devices are initialized as soon as their deps are ready

Generated async code expresses these as =.await= calls:

#+begin_src rust
// flash0 is child of fmc in config hierarchy
async fn init_flash0(services: &ServiceRegistry) -> Result<(), Error> {
    // Automatically waits for parent
    let fmc = services.get::<SpiController>(dev::FMC).await;
    
    let flash = spi::init_flash(&CONFIG.flash0, fmc)?;
    services.publish(dev::FLASH0, &flash);
    Ok(())
}
#+end_src

The minimal executor polls all futures. When =init_flash0= is polled before =init_fmc= completes, it returns =Pending=. Once =init_fmc= publishes its service, the next poll of =init_flash0= succeeds.

No topological sort needed. No explicit ordering in config. Dependencies are natural.

    pub const SILICON_REV: usize = 0x004;

    pub const CLK_STOP1: usize = 0x080;
    pub const CLK_STOP2: usize = 0x084;
    pub const RESET_CTRL1: usize = 0x040;
    pub const RESET_CTRL2: usize = 0x050;
    pub const HPLL_PARAM: usize = 0x200;
    pub const MPLL_PARAM: usize = 0x220;

    A0,
    A1, 
    A2,
    A3,

}

impl ClockService {
    /// Get HCLK frequency in Hz
    pub fn hclk_hz(&self) -> u32 {
        // Read HPLL config and calculate
        let hpll = self.read(reg::HPLL_PARAM);
        // ... calculation based on PLL settings
        200_000_000  // Simplified: 200MHz
    }
    
    /// Enable a clock gate
    pub fn enable_clock(&self, gate: ClockGate) {
        let (reg_offset, bit) = gate.reg_and_bit();
        let val = self.read(reg_offset);
        self.write(reg_offset, val & !(1 << bit));  // Clear bit to enable
    }
    
    /// Assert/deassert reset
    pub fn reset(&self, reset: ResetLine, assert: bool) {
        let (reg_offset, bit) = reset.reg_and_bit();
        let val = self.read(reg_offset);
        if assert {
            self.write(reg_offset, val | (1 << bit));
        } else {
            self.write(reg_offset, val & !(1 << bit));
        }
    }
    
    fn read(&self, offset: usize) -> u32 {
        unsafe { read_volatile((self.base + offset) as *const u32) }
    }
    
    fn write(&self, offset: usize, val: u32) {
        unsafe { write_volatile((self.base + offset) as *mut u32, val) }
    }
}

* Filesystem Implementation

** Overview

The filesystem stores:
1. *Fixed header* (=zerocopy=): Magic, version, offsets - parsed with zero overhead
2. *File table* (=postcard=): List of file entries with hashes - same structs as RON config
3. *Embedded config* (=postcard=): Optional runtime config blob - the =FstartConfig= struct serialized
4. *Signatures*: Cryptographic signatures over the above

The key point: =postcard= serializes the same Rust structs that =ron= deserializes. Build tool reads RON, firmware reads postcard, both use identical types.

** Filesystem Parser (for firmware)

#+begin_src rust
//! fstart-fs/src/parser.rs

use crate::{FsHeader, FileEntry, MAGIC, VERSION};
use fstart_driver::Error;

pub struct Filesystem<'a> {
    base: &'a [u8],
    header: &'a FsHeader,
    files: &'a [FileEntry],
}

impl<'a> Filesystem<'a> {
    /// Parse filesystem from memory-mapped flash
    pub fn from_bytes(data: &'a [u8]) -> Result<Self, Error> {
        if data.len() < core::mem::size_of::<FsHeader>() {
            return Err(Error::InvalidConfig);
        }
        
        // Safety: FsHeader is repr(C) and FromBytes
        let header = unsafe {
            &*(data.as_ptr() as *const FsHeader)
        };
        
        // Validate magic
        if header.magic != MAGIC {
            return Err(Error::InvalidConfig);
        }
        
        if header.version != VERSION {
            return Err(Error::InvalidConfig);
        }
        
        // Get file table
        let table_start = header.file_table_offset as usize;
        let table_end = table_start + header.file_table_size as usize;
        
        if table_end > data.len() {
            return Err(Error::InvalidConfig);
        }
        
        let file_count = header.file_count as usize;
        let files = unsafe {
            core::slice::from_raw_parts(
                data[table_start..].as_ptr() as *const FileEntry,
                file_count,
            )
        };
        
        Ok(Self { base: data, header, files })
    }
    
    /// Find a file by name
    pub fn find(&self, name: &str) -> Option<&FileEntry> {
        self.files.iter().find(|f| {
            let fname = core::str::from_utf8(&f.name)
                .unwrap_or("")
                .trim_end_matches('\0');
            fname == name
        })
    }
    
    /// Get file contents
    pub fn read(&self, entry: &FileEntry) -> &[u8] {
        let start = entry.offset as usize;
        let end = start + entry.stored_size as usize;
        &self.base[start..end]
    }
    
    /// Verify file hash
    pub fn verify_hash(&self, entry: &FileEntry) -> bool {
        use sha2::{Sha512, Digest};
        
        let data = self.read(entry);
        let mut hasher = Sha512::new();
        hasher.update(data);
        let computed = hasher.finalize();
        
        computed.as_slice() == &entry.hash
    }
}
#+end_src

** Loading Config at Runtime (postcard)

For "bulked" builds where config is determined at runtime, firmware deserializes the embedded config blob:

#+begin_src rust
//! Firmware: loading embedded config blob
use fstart_config::FstartConfig;  // Same struct as RON uses!

impl<'a> Filesystem<'a> {
    /// Load embedded config (postcard-serialized)
    pub fn load_config(&self) -> Result<FstartConfig, Error> {
        let entry = self.find("config.bin")
            .ok_or(Error::InvalidConfig)?;
        
        // Verify integrity first
        if !self.verify_hash(entry) {
            return Err(Error::InvalidConfig);
        }
        
        let data = self.read(entry);
        
        // Firmware deserializes the binary blob
        postcard::from_bytes(data)
            .map_err(|_| Error::InvalidConfig)
    }
}
#+end_src

The build tool writes this blob only when runtime config is needed - for static/"shredded" builds, the config is baked into generated code and no runtime deserialization occurs.

** Build Command Implementation

#+begin_src rust
//! fstart-tool/src/cmd/build.rs

use anyhow::{Context, Result, bail};
use fstart_config::FstartConfig;
use std::path::Path;
use std::process::Command;

pub fn build(
    config_path: &Path,
    output: &Path,
    key_path: Option<&Path>,
    no_build: bool,
) -> Result<()> {
    // 1. Load and validate config
    println!("Loading config: {}", config_path.display());
    let config = load_config(config_path)?;
    
    println!("Board: {} ({})", config.board.name, config.board.vendor);
    
    // 2. Generate code for each stage
    for stage in &config.boot.stages {
        println!("Generating bindings for stage: {}", stage.name);
        generate_stage_bindings(&config, stage)?;
    }
    
    // 3. Build stages
    if !no_build {
        for stage in &config.boot.stages {
            println!("Building stage: {}", stage.name);
            build_stage(&config, stage)?;
        }
    }
    
    // 4. Create filesystem image
    println!("Creating filesystem...");
    let mut image = create_image(&config)?;
    
    // 5. Sign
    let key = match key_path {
        Some(path) => {
            println!("Loading signing key: {}", path.display());
            load_signing_key(path)?
        }
        None => {
            println!("Generating ephemeral signing key");
            generate_ephemeral_key()?
        }
    };
    
    sign_image(&mut image, &key, &config.security)?;
    
    // 6. Write output
    std::fs::write(output, &image)?;
    println!("Wrote: {} ({} bytes)", output.display(), image.len());
    
    Ok(())
}

fn generate_stage_bindings(config: &FstartConfig, stage: &StageConfig) -> Result<()> {
    // Generate stages/{name}/src/generated.rs
    let gen_path = format!("stages/{}/src/generated.rs", stage.name);
    
    let code = codegen::generate_boot_flow(config, stage)?;
    std::fs::write(&gen_path, code)?;
    
    Ok(())
}

fn build_stage(config: &FstartConfig, stage: &StageConfig) -> Result<()> {
    let target = match &config.hardware.soc {
        SocType::AspeedAst2600(_) => "armv7a-none-eabi",
        _ => bail!("Unsupported SoC"),
    };
    
    let mut cmd = Command::new("cargo");
    cmd.arg("build")
        .arg("--release")
        .arg("--target").arg(target)
        .arg("-p").arg(format!("fstart-stage-{}", stage.name));
    
    for feature in &stage.features {
        cmd.arg("--features").arg(feature);
    }
    
    let status = cmd.status()?;
    if !status.success() {
        bail!("Cargo build failed for stage {}", stage.name);
    }
    
    Ok(())
}

* Crate Recommendations Summary

** Data Flow
#+begin_example
Human writes RON  -->  Build tool parses (ron crate)  -->  Rust structs in memory
                                                                  |
                                 +--------------------------------+--------------------------------+
                                 |                                |                                |
                                 v                                v                                v
                          Code generation               Write postcard blob              Direct use for
                          (generated.rs)                to flash image                   image layout
                                                        (only if runtime                 calculations
                                                        config needed)
#+end_example

Build tool works entirely with Rust structs. Serialization to postcard only happens when writing binary blobs that firmware will read at runtime.

* Open Questions

1. *Allocator*: Do we need heap allocation in firmware stages?
   - Option A: Static only (no Box, no Vec) - simplest
   - Option B: Bump allocator with fixed buffer - moderate
   - Option C: Full allocator - most flexible
   - *Recommendation*: Start with A, add B if needed

2. *Async executor details*:
   - Decided: Yes, use async for dependency-driven init ordering
   - Minimal executor (~50 lines) that just polls, never sleeps
   - No embassy dependency - we don't need timers/interrupts/wakers
   - If we later need timeouts (e.g., DRAM training), add targeted support

3. *Error handling in firmware*:
   - Option A: Panic on error (simplest, catches bugs)
   - Option B: Return errors, handle gracefully
   - *Recommendation*: A for development, B for production

4. *Config embedding*: How does firmware access config?
   - Option A: Generate Rust consts from config
   - Option B: Embed serialized config blob
   - *Recommendation*: A for shredded builds (current focus)

5. *Service lifetime management*:
   - Services are published to a global registry
   - Currently using raw pointers (single-threaded boot, no races)
   - May need to revisit for multi-core init if that becomes relevant
